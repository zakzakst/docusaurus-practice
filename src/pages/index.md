# Markdown page example

You don't need React to write simple standalone pages.

```
開発ドキュメントの役割は1つだけ

「あとから来た人が、迷わず動ける状態を作る」

未来の自分

新しく入ったメンバー

他チームの人

が 質問しなくても理解できる のが理想です。

👉
「全部を書く」より
「迷うポイントを潰す」 が大事です。
```

### 最低限そろえるべき資料
1. 概要・全体像（必須）：「このシステムは何か？」を一瞬で理解させる
    - 何のためのシステムか
    - 誰向けか
    - 何ができて、何ができないか
    - 全体構成図（あれば）
2. セットアップ手順：新人が詰まらずに環境構築できる
    - 必要な前提（Node / Docker / 権限など）
    - 初回セットアップ
    - よくあるエラーと対処法
3. 開発ルール・ガイドライン：コードの「暗黙知」をなくす
    - ブランチ運用
    - コーディング規約
    - commit / PRルール
    - ディレクトリ構成の意図
4. 機能・仕様ドキュメント：「なぜこうなっているか」を残す
    - 機能概要
    - 仕様
    - 制約
    - 例外ケース
5. 運用・保守：障害・変更に耐える
    - デプロイ手順
    - 環境差分
    - ログ・監視
    - 障害対応フロー

あと下記も欲しい
- テンプレート
- デザインドック
- ペルソナ
- カスタマージャーニー
- ユーザーストーリー
- 調査資料（マーケティングとか競合調査）
